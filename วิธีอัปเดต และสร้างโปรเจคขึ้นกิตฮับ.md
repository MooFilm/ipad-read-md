# วิธีอัปเดต และสร้างโปรเจคขึ้นกิตฮับ

ไฟล์นี้เขียนไว้สำหรับใช้งานจริงกับโปรเจกต์นี้โดยตรง เพื่อให้เข้าใจว่า:

1. Git คืออะไร
2. GitHub คืออะไร
3. เวลาสร้างโปรเจกต์ใหม่ขึ้น GitHub ต้องทำอะไรบ้าง
4. เวลามีการแก้โปรแกรมแล้วอยากอัปเดตเว็บ ต้องทำอะไรบ้าง
5. มีอะไรที่ต้องระวังเป็นพิเศษ
6. ถ้า error ต้องทำอย่างไร

เอกสารนี้พยายามอธิบายแบบภาษาคนใช้งานจริง ไม่เน้นศัพท์ยากเกินจำเป็น

---

## 1. Git คืออะไร

Git คือระบบบันทึกประวัติการเปลี่ยนแปลงของไฟล์ในโปรเจกต์

ให้คิดง่าย ๆ ว่า Git เป็นเหมือน:

- ระบบเซฟงานทีละรอบ
- ระบบย้อนดูว่าเราแก้อะไรไปบ้าง
- ระบบช่วยส่งงานขึ้น GitHub

ตัวอย่าง:

วันนี้คุณแก้หน้าเว็บ

- เพิ่มปุ่มใหม่
- แก้สี
- เพิ่มไฟล์ `.md`

Git จะช่วยบันทึกว่า “รอบนี้คุณแก้อะไร”

---

## 2. GitHub คืออะไร

GitHub คือเว็บที่เอาโปรเจกต์จาก Git ไปเก็บออนไลน์

ข้อดีคือ:

- เก็บสำรองโปรเจกต์ไว้บนอินเทอร์เน็ต
- เปิดจากเครื่องอื่นได้
- ใช้ GitHub Pages ทำเว็บให้เปิดบน iPad ได้
- มี GitHub Actions ช่วย deploy อัตโนมัติ

สรุปสั้น ๆ:

- `Git` = ระบบบันทึกงานในเครื่อง
- `GitHub` = ที่เก็บโปรเจกต์ออนไลน์

---

## 3. หลักการง่ายที่สุดของการอัปเดตโปรเจกต์

ทุกครั้งที่คุณแก้โปรเจกต์แล้วอยากให้เว็บอัปเดตบน GitHub Pages ให้คิดเป็น 4 ขั้นตอน:

1. แก้ไฟล์ในเครื่อง
2. บอก Git ว่าไฟล์ไหนเปลี่ยน
3. บันทึกรอบการเปลี่ยนแปลง
4. ส่งขึ้น GitHub

คำสั่งหลักที่ใช้บ่อยคือ:

```powershell
git add .
git commit -m "ข้อความอธิบายรอบนี้"
git push
```

---

## 4. ความหมายของคำสั่งหลัก

### 4.1 `git add .`

แปลว่า:

- เอาไฟล์ที่เปลี่ยนทั้งหมดในโฟลเดอร์นี้มาเตรียมส่ง

ถ้าคุณ:

- แก้ `src/App.jsx`
- แก้ `src/styles.css`
- เพิ่มไฟล์ `.md` ใหม่

คำสั่งนี้จะเตรียมไฟล์เหล่านั้นไว้

### 4.2 `git commit -m "..."`

แปลว่า:

- บันทึกประวัติการเปลี่ยนแปลงรอบนี้

ตัวอย่าง:

```powershell
git commit -m "improve reader UI"
```

หรือ

```powershell
git commit -m "add new markdown files"
```

### 4.3 `git push`

แปลว่า:

- ส่ง commit ที่บันทึกแล้วจากเครื่องขึ้น GitHub

หลังจาก `git push`:

- GitHub Actions จะทำงาน
- GitHub Pages จะ deploy เว็บใหม่
- หน้าเว็บบน iPad จะอัปเดต

---

## 5. ขั้นตอน “สร้างโปรเจกต์ขึ้น GitHub” ครั้งแรก

อันนี้ทำครั้งแรกครั้งเดียว

### 5.1 สร้าง repository บน GitHub

1. เข้า GitHub
2. กด `New repository`
3. ตั้งชื่อ repo
4. เลือก `Public`
5. กด `Create repository`

### 5.2 เชื่อมโปรเจกต์ในเครื่องกับ GitHub

ใน PowerShell ภายในโฟลเดอร์โปรเจกต์:

```powershell
git init
git add .
git commit -m "initial project"
git branch -M main
git remote add origin https://github.com/USERNAME/REPO.git
git push -u origin main
```

หมายเหตุ:

- `USERNAME` คือชื่อบัญชี GitHub
- `REPO` คือชื่อ repository

ตัวอย่าง:

```powershell
git remote add origin https://github.com/MooFilm/ipad-read-md.git
```

## 6. ขั้นตอน “อัปเดตโปรเจกต์” ทุกครั้งหลังแก้ไฟล์

อันนี้คือขั้นตอนที่คุณจะใช้บ่อยที่สุด

### วิธีมาตรฐาน

เมื่อคุณแก้โปรเจกต์เสร็จแล้ว:

```powershell

git push
```

ตัวอย่าง:

```powershell
git add .
git commit -m "improve floating bookmark UI"
git push
```

หรือถ้าเพิ่มไฟล์ `.md` ใหม่:

```powershell
git add .
git commit -m "add new markdown books"
git push
```

หลังจากนั้น:

1. รอ GitHub Actions deploy
2. เปิดลิงก์เว็บ
3. รีเฟรชบน iPad

## 7. ถ้าคุณแก้แค่ไฟล์บางไฟล์

บางครั้งคุณไม่อยาก `git add .` ทั้งหมด เพราะอยากเลือกเฉพาะบางไฟล์

ตัวอย่าง:

```powershell
git add src/App.jsx src/styles.css
git commit -m "refine reader layout"
git push
```

กรณีนี้เหมาะเมื่อ:

- อยากอัปเฉพาะโค้ด
- ยังไม่อยากส่งไฟล์อื่นขึ้นไป

---

## 8. ถ้าคุณเพิ่มไฟล์ `.md` สำหรับอ่านบนเว็บ

ให้เอาไฟล์ไปไว้ใน:

```text
public/docs
```

หรือในโฟลเดอร์ย่อย เช่น:

```text
public/docs/thesis/chapter-1.md
public/docs/lecture/week-1.md
public/docs/research/paper-notes.md
```

จากนั้นรัน:

```powershell
git add .
git commit -m "add new markdown files"
git push
```

พอ deploy เสร็จ ไฟล์จะขึ้นในหน้าเว็บอัตโนมัติ

## 9. ตอนนี้โปรเจกต์นี้รองรับโฟลเดอร์ย่อยแล้ว

คุณสามารถจัดไฟล์ได้แบบนี้:

```text

public/docs/thesis/chapter-2.md
public/docs/lecture/week-1.md

```

หน้าแรกของเว็บจะจัดเป็นหมวดตามโฟลเดอร์ให้เอง

เช่น:

- `thesis`
- `lecture`
- `research`

## 10. หลัง `git push` แล้วเกิดอะไรขึ้น

หลังจากคุณ `git push`

GitHub จะทำสิ่งต่อไปนี้อัตโนมัติ:

1. รับโค้ดใหม่ขึ้น GitHub
2. GitHub Actions จะเริ่มทำงาน
3. ระบบ build เว็บใหม่
4. ระบบ deploy ไปที่ GitHub Pages
5. เว็บที่ลิงก์เดิมจะอัปเดต

ลิงก์เว็บของโปรเจกต์นี้คือ:

[https://moofilm.github.io/ipad-read-md/](https://moofilm.github.io/ipad-read-md/)

ดังนั้นส่วนใหญ่คุณไม่ต้องไปกดอะไรเพิ่มบน GitHub

แค่:

- `git push`
- รอ
- รีเฟรชเว็บ

## 11. จะเช็กได้อย่างไรว่า deploy สำเร็จหรือยัง

ไปที่แท็บ `Actions` ใน GitHub repo

ถ้าเห็น workflow ล่าสุดเป็นสีเขียว:

- แปลว่า deploy ผ่านแล้ว

ถ้าเป็นสีแดง:

- แปลว่า deploy fail
- ต้องกดเข้าไปดู error

โดยปกติถ้า `git push` ผ่านและ workflow เขียว เว็บจะอัปเดตเอง

## 12. ควรระวังอะไรบ้างทุกครั้งก่อนอัปขึ้น GitHub

นี่คือส่วนสำคัญมาก

### 12.1 อย่าอัปไฟล์ลับขึ้นไป

ตัวอย่างไฟล์ลับ:

- `.env`
- `.env.local`
- `secret.key`
- `private.pem`
- token หรือ API key ที่เขียนค้างไว้ในไฟล์

ตอนนี้โปรเจกต์นี้ตั้ง `.gitignore` กันไว้แล้วสำหรับ:

- `.env`
- `.env.*`
- `*.pem`
- `*.key`
- `*.p12`
- `*.pfx`

ดังนั้นไฟล์พวกนี้จะไม่ถูก push ง่าย ๆ

### 12.2 อย่าใส่รหัสลับไว้ในโค้ด

ตัวอย่างที่ไม่ควรทำ:

```js
const apiKey = "123456SECRET"
```

ถ้าใส่แบบนี้ ต่อให้ไม่มี `.env` ข้อมูลก็ยังหลุดได้

### 12.3 เช็กก่อนว่าไฟล์ไหนจะถูกส่ง

ใช้คำสั่ง:

```powershell
git status --short
```

คำสั่งนี้จะบอกว่า:

- ไฟล์ไหนถูกแก้
- ไฟล์ไหนเพิ่มใหม่
- ไฟล์ไหนกำลังรอ commit

ก่อน `git add .` ถ้าเห็นไฟล์แปลก ๆ ให้หยุดเช็กก่อน

---

## 13. ถ้า `.env` หลุดขึ้น GitHub จะเกิดอะไรขึ้น

ถ้า `.env` หลุดขึ้น GitHub และข้างในมี:

- API key
- token
- password
- database URL

ผลที่อาจเกิดขึ้นคือ:

- คนอื่นเอา key ไปใช้ได้
- ระบบโดนคิดเงิน
- ข้อมูลรั่ว
- ต้องรีบเปลี่ยน secret ทั้งหมด

หลักคิดสำคัญ:

> ถ้าไฟล์ลับหลุดขึ้น GitHub ให้ถือว่า secret นั้นใช้ต่อไม่ได้แล้ว

ต้อง:

1. เปลี่ยน secret ใหม่
2. revoke ของเดิม
3. ลบไฟล์ออก
4. ถ้าร้ายแรงอาจต้องล้าง git history

โชคดีคือโปรเจกต์นี้ตอนนี้ยังไม่พบ `.env` หลุด

---

## 14. วิธีเช็กว่า `.env` ถูกอัปขึ้นไปหรือยัง

ในเครื่อง ใช้:

```powershell

```

และใช้:

```powershell
git status --short
```

เพื่อดูว่ามีไฟล์ลับค้างอยู่หรือไม่

## 15. ถ้า `git push` ไม่ผ่าน ต้องทำอย่างไร

### 15.1 กรณีขึ้นว่า `fetch first`

ตัวอย่าง error:

```text
Updates were rejected because the remote contains work that you do not have locally
```

แปลว่า:

- ฝั่ง GitHub มี commit ใหม่
- แต่ในเครื่องยังไม่มี

วิธีแก้:

```powershell
git pull --rebase origin main
git push
```

### 15.2 กรณีมี conflict

ถ้าไฟล์เดียวกันถูกแก้ทั้งในเครื่องและบน GitHub พร้อมกัน อาจเกิด conflict

ให้:

1. ดูว่าไฟล์ไหน conflict
2. เลือกว่าจะใช้เนื้อหาแบบไหน
3. แก้ไฟล์ให้เรียบร้อย
4. `git add .`
5. `git rebase --continue` หรือ `git commit`

ถ้าไม่แน่ใจ ให้หยุดก่อนแล้วตรวจทีละไฟล์

### 15.3 กรณี repo ไม่เจอ

ตัวอย่าง:

```text
Repository not found
```

แปลว่า:

- ชื่อ repo ผิด
- URL remote ผิด
- หรือกำลังใช้บัญชี GitHub ผิด

ใช้เช็ก:

```powershell
git remote -v
```

---

## 16. วิธีเช็กว่า remote ถูกต้องไหม

ใช้:

```powershell
git remote -v
```

สำหรับโปรเจกต์นี้ควรเห็นประมาณ:

```text
origin  https://github.com/MooFilm/ipad-read-md.git (fetch)
origin  https://github.com/MooFilm/ipad-read-md.git (push)
```

ถ้าผิด ให้แก้ด้วย:

```powershell
git remote set-url origin https://github.com/MooFilm/ipad-read-md.git
```

## 20. ถ้าอยากอัปไฟล์ผ่าน GitHub หน้าเว็บแทนคอม

2. เข้าโฟลเดอร์ `public/docs`
3. กด `Add file`

5. commit บนหน้าเว็บ

- ถ้าคุณอัปไฟล์จากหน้าเว็บ แล้วกลับมาแก้ในเครื่องต่อ

```powershell

```
